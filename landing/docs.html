Quickstart

One command sets up everything: detects your browsers, installs the Chrome extension, finds AI agents on your machine, and configures MCP.

npx hanzi-in-chrome setup

The setup wizard supports Claude Code, Cursor, Windsurf, Claude Desktop, and Codex. It auto-detects which agents are installed and configures each one.

What the setup does

  • 1. Checks for the Chrome extension — opens the install page if missing
  • 2. Scans for supported AI agents on your machine
  • 3. Adds Hanzi as an MCP server to each agent's config
  • 4. Optionally imports credentials (Claude Code OAuth, Codex, or API key)

What the CLI sets up

The setup wizard configures Hanzi to use your own model credentials (BYOM). You provide a Claude, GPT, or Gemini API key. The agent runs locally — no data leaves your machine.

What the CLI does today Bring your own model

Auto-detects Claude Code and Codex credentials, or prompts for an API key or custom endpoint. This is the current self-serve path.

Separate path Managed access (by request)

We handle model routing — no API key needed. This is a separate onboarding path, not yet part of the CLI. Contact us to get set up.

Supported credentials (BYOM)

  • Claude Code OAuth — auto-detected if you've run claude login
  • Codex — auto-detected if you've run codex login
  • API key — set ANTHROPIC_API_KEY or enter directly during setup
  • Custom endpoint — any OpenAI-compatible API (Ollama, LM Studio, etc.)

Manual setup

If you prefer to set things up yourself:

  • 1. Install the Chrome extension
  • 2. Add the MCP server to your agent:
    Claude Code: claude mcp add browser -- npx -y hanzi-in-chrome
    Others: add {"command":"npx","args":["-y","hanzi-in-chrome"]} to your MCP config
  • 3. Provide credentials (see above)

Test it

After setup, ask your agent something that requires a browser:

"Go to Hacker News and tell me the top 3 stories right now"

Hanzi opens a browser tab, reads the page, and returns the answer inside your agent session.

Troubleshooting

Extension not detected

Make sure the Chrome extension is installed and enabled. Reload the extension at chrome://extensions if needed.

Agent can't find Hanzi

Restart your AI agent after running setup. The MCP config is written to disk but agents need a restart to pick it up.

Task fails or times out

Check that your credentials are valid. Verify your API key works with your provider directly. If using a custom endpoint, confirm it's reachable.

Build with Hanzi

Developer quickstart

Integrate browser automation into your product. Your app calls the Hanzi API, a real browser executes the task, and you get the result back.

How it works

  • 1. Get an API key — sign in, then create a key via POST /v1/api-keys with your session cookie
  • 2. Pair a browser — create a pairing token, have your user install the extension and paste the token in Settings → Managed tab
  • 3. Run a task — call the API with a task description and a browser session ID
  • 4. Get the result — poll the task or use runTask() which waits for completion

TypeScript SDK

The SDK source lives in sdk/ in the repo. It is not yet published to npm — clone the repo and install from sdk/ directly.

import { HanziClient } from '@hanzi/browser-agent';

const client = new HanziClient({
  apiKey: process.env.HANZI_API_KEY,
});

// 1. Create a pairing token for your user
const { pairingToken, expiresAt } = await client.createPairingToken();
// → show pairingToken in your UI for the user to enter in the extension

// 2. List connected browser sessions
const sessions = await client.listSessions();
const session = sessions[0];

// 3. Run a task
const result = await client.runTask({
  browserSessionId: session.id,
  task: 'Read the patient chart on the current page',
});

console.log(result.answer);

REST API

Most endpoints accept either an API key (Authorization: Bearer hic_...) or a Better Auth session cookie. Sign in first, then create a key via the API.

GET /v1/health Health check (no auth required)
POST /v1/api-keys Create an API key for your workspace. Body: {"name":"my-key"}. Key shown once.
GET /v1/api-keys List API keys (shows prefix only, not full key)
DELETE /v1/api-keys/:id Delete an API key
POST /v1/browser-sessions/pair Create a pairing token. Optional body: {"label":"...", "external_user_id":"..."}
POST /v1/browser-sessions/register Exchange pairing token for a session credential
GET /v1/browser-sessions List sessions with status, label, and external_user_id
POST /v1/tasks Start a browser automation task
GET /v1/tasks/:id Get task status and result
POST /v1/tasks/:id/cancel Cancel a running task
GET /v1/usage Usage summary for your workspace

Browser pairing model

Your app creates a pairing token (valid for 5 minutes). The end user installs the Hanzi extension, opens Settings → Managed tab, and pastes the token. The extension connects to the hosted Hanzi backend automatically — no URL entry needed. Once paired, you get a persistent browser session tied to that browser. Tasks run against a specific session — each user's browser is isolated.

Sessions have a 30-day token expiry. If the extension disconnects (browser restart, network drop), it auto-reconnects using the stored session token. No re-pairing needed.

Session metadata

When creating a pairing token, you can attach a label and external_user_id to identify whose browser this is in your system. This metadata is inherited by the browser session and returned in GET /v1/browser-sessions. Use it to map Hanzi sessions to your users.

Session lifecycle

Pairing

Create token → user enters it → session registered

Your backend creates a pairing token. Your UI shows it to the user. The user pastes it in the Hanzi extension (Settings → Managed tab → Connect). The extension exchanges it for a session.

Connected

Run tasks → get results

While the browser is open and the extension connected, you can run tasks against the session. Each task executes in the user's real browser.

Disconnected

Auto-reconnect on browser restart

If the browser closes, the session is preserved. When the browser reopens, the extension reconnects automatically. Tasks queued during disconnect will fail — check session status first.

Concepts

All modes

Browser sessions

A browser session represents one connected Chrome instance. Tasks execute against a specific session, not a random tab.

All modes

Tools

The agent uses tools like read_page, computer, navigate, and form_input to interact with the browser.

API / SDK

Workspaces

A workspace groups browser sessions, tasks, API keys, and usage. Each account gets a default workspace.

All modes

MCP tools

browser_start runs a task, browser_message sends follow-ups, browser_status checks progress, browser_stop cancels, browser_screenshot captures the page.

Security

All modes

Privacy

BYOM: no data leaves your machine. Managed/API: tasks processed server-side with hashed credentials. Full privacy policy.

API / SDK

Authentication

API keys (hashed at rest), pairing tokens (5-min expiry), session tokens (30-day expiry, auto-rotated).
